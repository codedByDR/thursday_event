1. System Overview

STHENIA is a modern, AI-powered web application built using a microservices-inspired architecture with clear separation of concerns:

Core Architecture Layers

  • Frontend: React 18 SPA with TypeScript
  • Backend: FastAPI REST API with async Python
  • AI Layer: LangGraph multi-agent system
  • Data Layer: SQLite database + ChromaDB vector store
  • Cache Layer: Redis for sessions

3. Component Details

3.1 Frontend Architecture

Technologies

  • React 18 with TypeScript
  • Vite for build tooling
  • Tailwind CSS for styling
  • Zustand for state management
  • TanStack Query for server state
  • Framer Motion for animations

Key Components

  • Pages: Login, Dashboard, Requests, Wellbeing
  • Components: Kira Widget, Request Wizard, Analytics Charts
  • Stores: Auth Store (persistent)
  • Services: API client with interceptors

Design Pattern: Component-based architecture with hooks

3.2 Backend Architecture

Technologies

  • FastAPI for REST API
  • SQLAlchemy 2.0 (async ORM)
  • Pydantic for validation
  • JWT for authentication
  • APScheduler for background jobs

Layers:

  1. API Layer (app/api/routes/)
    • Route handlers
    • Request/response models
    • Dependency injection
  2. Service Layer (app/services/)
    • Business logic
    • Data transformations
    • External integrations
  3. Data Layer (app/models.py, app/db/)
    • ORM models
    • Database sessions
    • Migrations
  4. Core Layer (app/core/)
    • Configuration
    • Security utilities
    • Logging
    • Scheduler

Design Pattern: Layered architecture with dependency injection

3.3 AI Agent Architecture

Framework: LangGraph for stateful multi-agent orchestration

Agent Types:

  1. Request Agent: Main workflow orchestration
  2. Justification Generator: Text transformation
  3. Counselling Agent: Mental health support
  4. Burnout Detection: Risk assessment
  5. POSH Agent: Harassment reporting
  6. Shift Safety: Safety evaluation
  7. Maternity Return: Return-to-work planning
  8. Meal Planner: Nutrition planning
  9. Grocery Reminder: Shopping assistance
  10. Alarm & Reminder: Time management
  11. Daily Todo: Task management
  12. Wellbeing Tracker: Health monitoring
  13. Micro Learning: Skill development
  14. Policy Checker: Document verification

Agent Communication

  • Stateful workflow using LangGraph StateGraph
  • Shared state across nodes
  • Conditional branching
  • Human-in-the-loop checkpoints

3.4 Data Architecture

Primary Database: SQLite 3

  • 24 tables
  • Full ACID compliance
  • WAL mode for concurrent reads
  • Async operations via aiosqlite

Vector Store: ChromaDB

  • Policy document embeddings
  • Semantic search
  • RAG retrieval

Cache Layer: Redis

  • Session management
  • 10-minute TTL
  • Token blacklist

Storage

  • Policy documents: data/policies/
  • Database file: data/sthenia.db
  • Logs: logs/backend.log

4. Security Architecture

4.1 Authentication Flow

1. User submits credentials
   ↓
2. FastAPI validates credentials
   ↓
3. Generate JWT access token (15 min)
   ↓
4. Generate JWT refresh token (7 days)
   ↓
5. Return tokens to client
   ↓
6. Client stores tokens (local storage)
   ↓
7. Include access token in Authorization header
   ↓
8. Verify token on each request
   ↓
9. Refresh access token as needed

4.2 Authorization

Role-Based Access Control (RBAC)

  • Employee: Own data + create requests
  • Supervisor: Approve team requests + view team data
  • HR: View all requests + analytics + harassment reports
  • Admin: Full access + policy management

Implementation

  • Dependency injection in route handlers
  • require_role() decorator
  • Permission checks in service layer

4.3 Data Encryption

  • Passwords: bcrypt (cost factor 12)
  • Sensitive fields: AES-256
  • HTTPS enforced in production
  • JWT signed with HS256

4.4 Security Headers

  • X-Frame-Options: DENY
  • X-Content-Type-Options: nosniff
  • X-XSS-Protection: 1; mode=block
  • Strict-Transport-Security: max-age=31536000

8. API Design

8.1 RESTful Endpoints

Authentication:

  • POST /api/auth/register
  • POST /api/auth/login
  • POST /api/auth/refresh
  • POST /api/auth/logout

Requests:

  • GET /api/requests/
  • POST /api/requests/
  • GET /api/requests/{id}
  • PUT /api/requests/{id}

Approvals:

  • GET /api/approvals/
  • POST /api/approvals/{id}/approve
  • POST /api/approvals/{id}/reject

Kira:

  • POST /api/kira/chat
  • GET /api/kira/greeting

8.2 Response Format

{
  "status": "success",
  "data": { ... },
  "message": "Operation completed"
}

8.3 Error Format

{
  "detail": "Error message",
  "status_code": 400
}

9. Technology Decisions

9.1 Why SQLite?

  • Zero configuration
  • Single file portability
  • Sufficient for single-org
  • Easy to migrate later

9.2 Why LangGraph?

  • Stateful workflows
  • Human-in-the-loop support
  • Multi-agent orchestration
  • Conditional branching

9.3 Why FastAPI?

  • Async support
  • Auto-generated docs
  • Type safety with Pydantic
  • High performance

9.4 Why React + Vite?

  • Fast development
  • Modern build tooling
  • TypeScript support
  • Rich ecosystem
